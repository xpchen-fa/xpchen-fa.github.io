窗口函数

基本语法：
<窗口函数> over (partition by <用于分组的列名> order by <用于排序的列名>)
窗口函数位置可放：

  • 专用窗口函数，包括rank，dense_rank,row_number等；
  • 聚合函数，如sum，avg，count，max，min
  • 和错行函数（lead、lag）联合使用
  • 其他函数，first_value 和 last_value, ntile,ration_to_report等

专用窗口函数

  • rank函数：计算排序时，如果存在相同位次的记录，则会跳过之后的位次
  • dense_rank函数：计算排序时，即使存在相同位次的记录，也不会跳过之后的位次
  • row_number函数：赋予唯一的连续位次

错行函数

  • lead(x,offset,default_value) 窗口内x列某行之后offset行的值，如果不存在该行，则取default_value
  • lag(x,offset,default_value) 窗口内x列某行之前offset行的值，如果不存在改行，则取default_value

HQL优化

  1. 使用分区剪裁、列剪裁
  • 在select中，只拿需要的列，如果有，尽量使用分区过滤，少用select *
  • 尽量提前做过滤，减少后续处理的数据量。例如两张表做join，join后再过滤与join前就做过滤，虽然结果可能一样，但shuffle的数据量会差别很大，导致前者执行效率较低。
  1. 虽然hive在做逻辑判断时会自动做类型转换，但可能有坑，例如 string 和 bigint 类型的值做比较的时候，实际上hive会把两个值都转成 double 再比较，而浮点数的相等判断是有一定误差的。所以最好事先手动做强制类型转换。
    另外，平时建表的时候，建议整数都用 bigint，不要求精准的浮点数用 double，字符串用string，要求精准的浮点数（例如金额等）用decimal。
  2. 少用多重count distinct
  1. 是否存在多对多关联
  2. 尽量原子化操作，尽量避免一个sql包含复杂逻辑
  3. 使用动态分区
    set hive.exec.dynamic.partition=true
    set hive.exec.dynamic.partition=nonstrict
  4. 通过执行计划可以看到具体在那里过滤，不确定就分析执行计划，看看执行时怎么样被优化的
  5. 数据倾斜
    首先定位倾斜的位置。一般是shuffle的时候才会发生倾斜，sql简单的情况下关注group by、join 部分即可，sql复杂时需要用 explain 语句来检查发生倾斜的stage对应的具体逻辑。
  6. group by 的倾斜处理
    group by 过程中发生的倾斜，可以配置 set hive.groupby.skewindata=true; 通过二次reduce来缓解倾斜。
