性能优化

性能优化大集合

Posted by Xiuping Chen on 2021-07-28

一、 网络传输

1. 开启HTTP/2

  • 多路复用：多个请求复用一次网络连接，实现真正意义的并发
  • 二进制传输：解析更高效、体积更小、不易出错
  • 消息头压缩：压缩消息头体积，建立”字典”，用索引号表示重复的字符串
  • 服务端推送：在用户请求HTML时，一并把页面其他资源主动推送过来（一般不开启）

2. 使用CDN静态部署

  • 全局负载均衡

  • 缓存系统（命中率—越高越好 & 回源率—越低越好，一般不高于5%）

  • 缓存策略调优：

    客户端：强缓存，协商缓存

    CDN节点：缓存时间、缓存刷新、缓存预热

3. 使用DNS预解析

DNS解析时比较耗时的操作，一些比较小的网站域名解析甚至需要0.5s之久

4. 提前建立网络连接

浏览器在建立网络连接时，要经过DNS解析、TCP握手等过程，在https场景下还需要进行TLS加密信息验证。我们可以使用preconnect的link标签提前触发上述操作

5. 网络传输

  • 域名收敛：把页面资源部署在尽可能少的域名下
  • 域名发散：把页面资源部署在尽可能多的域名下

随着时代发展域名发散逐渐落后，使用域名收敛配合H2，可以最大化的节省DNS解析、TCP建连等网络成本，更好的发挥其多路复用的优势，大幅提升页面性能。

6. 使用brotli压缩

谷歌在2015年发布brotli压缩算法，相比gzip，它具有更高的压缩比和更快的压缩性能。

  • javascript 体积比gzip小 14%
  • HTML 体积比gzip小 21%
  • CSS 体积比gzip小 17%

7. 优化HTTPS

目前整个互联网都在大力推行https，但是https的多次握手、TLS相关算法运算等会造成额外的开销导致未经优化的https的性能相比http差距较大

  1. Session Resume：把昂贵的计算结果保存在Session中复用
  2. OCSP Stampling：由服务端直接返回OCSP内容，而不需要客户端额外查询
  3. 使用TLS 1.3：只需要一次往返（1-RTT）就可以完成握手
  4. TCP Fast Open：一个RTT的时间内syn包发出的同时捎上应用层的数据
  5. HSTS：强制客户端（如浏览器）使用HTTPS与服务器创建连接

2. 宿主浏览器缓存

  • Memory 缓存
  • Cache API 缓存
  • HTTP 缓存 （强缓存/协商缓存）
  • Push 缓存：当我们使用H2 的Server Push功能时，浏览器会生成一层push缓存。这个缓存存在感比较弱，当其他形式的宿主缓存都失效时，才会使用push缓存。

当用户重复请求资源时，会按照一定的层次顺序来获取缓存资源，若当前层资源可用则立即返回，否则寻找下一层缓存。

Memory Cache > Cache API > HTTP Cache > Push Cache > Server

3. 运行时缓存

  • Cookie：存放一些简单数据
  • LocalStrotage： 存储具有一定体积的数据
  • DB：存储大量结构化数据
  • Memory：存放临时数据

三、资源网络加载

1. 资源预加载

  • 预加载当前页面的资源

    我们可以使用link preload 来声明当前页面所需要的核心资源，可以时一些关键字体、样式、脚本等资源：

​ 在不支持使用link preload的宿主环境中，我们可以通过js编码来提前发起资源请求

  • 预加载下级页面的资源
    我们可以使用link prefetch来声明跳转后的页面所需要的核心资源
    当用户停留在当前页面时，浏览器会在合适的空闲时机提前加载跳转后所需的资源，提升用户后续浏览的体验。

​ 我们可以使用link prerender来声明跳转后的页面所需要的核心资源

2. HTML加载

控制HTML体积：

  • 传输体积不要超过30KB
  • 减少DOM节点和属性
  • 压缩HTML代码
  • 谨慎使用内联

四、代码实际运行

  1. HTML语义化
  2. 多使用伪元素，减少js多DOM的查找遍历
  3. 能用HTML/CSS实现的效果就不要用JS
  4. 逻辑与展示节藕，避免不必要的JS引擎启动
  5. 减少作用域查找和闭包，避免==，使用块级作用域
